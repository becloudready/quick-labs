Data Platform Evolution

Four Eras, Four Problems They Solved

Each era didn't kill the previous — it added a layer. Most large organizations run all four today.

Era Roughly Built to solve
Operational Database (OLTP) 1970s – present Reliably record what's happening now — orders, inventory, accounts
Data Warehouse (OLAP) Late 1980s – present Answer business questions over years of data without slowing operations
Data Lake ~2010 – present Store anything (JSON, images, logs), cheaply, at unlimited scale
Lakehouse ~2019 – present Get ACID + transactions + governance back, on top of the cheap lake
Most large orgs run all four tiers simultaneously. The operational DB feeds the warehouse via ETL; raw events land in the lake; governed tables live in the lakehouse.

Era 1 — Operational Database

The Database: Built for Transactions, Not Analytics

What it does well

  • ACID transactions — every write is safe
  • Point lookups by primary key in milliseconds
  • Row updates and deletes
  • Normalized schemas, foreign-key constraints
Examples:

PostgreSQL, MySQL, Oracle, SQL Server, DynamoDB

Why it stops scaling for analytics

  • Row-oriented storage: query needing 3 columns out of 50 still reads the full row off disk
  • Single-machine compute: "Add more CPU" hits a ceiling; sharding is painful
  • Coupled storage + compute: pay for compute even with zero queries; can't scale independently
  • Indexes only help if you predicted the query at table-design time

The problem in practice

Running this query against a 2-billion-row Postgres table:

SELECT region, AVG(price)
FROM orders
WHERE year = 2024
GROUP BY region;
This query will be slow or impossible on a standard Postgres table of that size. The engine reads every column on every row to extract just region, price, and year — even though those are 3 of 50 columns.

The issue isn't that Postgres is bad — it's the wrong tool for this workload. It was designed for thousands of small, frequent writes; not one query over billions of rows.

Era 2 — Data Warehouse

The Data Warehouse: Re-Architected for Analytics

What changed

  • Columnar storage: each column stored together. 3/50 columns → ~3/50 of the data read. Compression dramatically better.
  • Massively Parallel Processing (MPP): query split across many compute nodes, results combined
  • Schema-on-write: data cleaned, validated, modeled into star/snowflake schemas before it lands
Examples:

Teradata, Redshift, Snowflake, BigQuery, Synapse

Where it still hurts

Pain point Why it matters
Cost at scale Per-TB pricing is high; 50TB of monthly logs is wasteful
Schema-on-write is rigid New JSON shape? Re-model, re-ETL history, re-deploy pipeline
Proprietary formats Snowflake data is in Snowflake — migrating means re-exporting everything
Coupled storage + compute On-prem DWs bundle them — hard to scale each independently
No unstructured story Images, audio, raw JSON don't model into relational schemas

The Key Architectural Shift

Separate Storage from Compute

Hadoop genesis (mid-2000s)

HDFS + MapReduce on the same commodity nodes. "Move the compute to the data." Storage and compute still scaled together — need 10× more storage? Buy 10× more nodes including CPU you don't need.

The cloud + S3 realization (2010s)

Network got fast and cheap. S3 = effectively unlimited capacity, near-zero idle cost. Spark, Presto, Athena, Trino, Snowflake — query engines that don't own their data.

Economic punchline: 100 TB of cold log data on S3: ~$2,300/month, no compute. Same 100 TB on Redshift cluster: easily $20,000+/month even if queried once a quarter.

Plus: data is liberated from the engine — Athena for SQL, EMR for ML, DuckDB for vectorized analytics, Lambda+DynamoDB for real-time API — all pointing at the same S3 path.

Model Storage cost (idle) Compute cost (idle) Compute (running)
Postgres / RDS Included in instance 24/7 even with zero queries Included
On-prem Teradata $$$ — bundled 24/7 even with zero queries Included
Hadoop (HDFS+MR) Cheap per-GB on owned hardware 24/7 — same nodes Included
Snowflake / BigQuery Cheap per-GB on cloud storage $0 (warehouse can be suspended) $$ per second / per TB scanned
S3 + Athena $0.023/GB-month $0 (no infrastructure) $5 per TB scanned

Era 3 — Data Lake

The Data Lake: Cheap Storage on S3, Compute Attached on Demand

The architectural move

Separate storage from compute. Put the data in object storage. Pick whichever query engine matches the workload.

What this buys

  • Cost: S3 is ~$0.023/GB/month. Athena charges only for what you scan.
  • Scale: Functionally unlimited. AWS itself stores trillions of objects.
  • Open formats: Parquet, ORC, Avro, JSON, CSV — readable by many tools.
  • Multi-engine: Athena, Redshift Spectrum, EMR, Glue ETL, Trino, DuckDB — all query the same S3 files.
  • Schema-on-read: land raw data first; infer or impose schema at query time.

Examples: AWS S3 + Athena/Glue/EMR · Azure ADLS + Synapse · GCP Cloud Storage + BigQuery External Tables

The catch — "data swamp"

  • No transactions: a failed Spark write leaves half-files; concurrent writers corrupt each other
  • No deletes/upserts at the file layer — you rewrite whole partitions
  • No schema enforcement — adding a column silently breaks downstream consumers
  • Hard to know "current state" when 10,000 Parquet files have been appended over 5 years
The data swamp problem: teams discovered that "land raw data first" without governance meant months later no one knew which files were authoritative, schemas had drifted silently, and one bad write had corrupted a partition that was still being read downstream.

Era 4 — Lakehouse

The Lakehouse: ACID on Object Storage via Open Table Formats

The fix

Add a metadata layer (Iceberg / Delta Lake / Hudi) on top of S3 that tracks which Parquet files belong to which version of which table.

What the metadata layer gives back

  • ACID transactions — atomic commits
  • Upserts and deletes at the row level
  • Schema evolution — add/rename/drop columns without breaking readers
  • Time travel — query the table as of yesterday / last week / before that bug
  • Concurrent writers — multiple jobs can write to the same table safely

All while keeping data in open Parquet files on S3, queryable by any compatible engine.

Examples: Apache Iceberg (Netflix-born, AWS-native) · Delta Lake (Databricks-led) · Apache Hudi (Uber-born, stream-write focused)

Warehouse vs Lake vs Lakehouse

Warehouse Lake Lakehouse
ACID transactions ✅ ❌ ✅
Multi-engine reads ❌ ✅ ✅
Cheap S3 storage ❌ ✅ ✅
Schema enforcement ✅ ❌ ✅
Open file format ❌ ✅ ✅

Why Teams Moved

Why Teams Moved to S3 in the First Place

# Force Impact
1 Cost: S3 storage ~10× cheaper than DW-managed storage at same scale Letting 5 years of logs sit "just in case" became economically rational
2 Scale beyond what a warehouse cluster could hold — trillions of events, petabytes of raw files Warehouses had hard ceilings; S3 had none
3 Semi-structured / unstructured data exploded — JSON event streams, images, audio, scraped HTML The warehouse schema-on-write model couldn't model it fast enough
4 Open formats matured — Parquet (2013), then Iceberg/Delta (2018+) "Lock-in" stopped being a reason to stay in proprietary DWs
5 Decoupled compute became viable — Spark, Presto/Trino, Athena, Spectrum all query S3 You could keep cold data cheap and rent compute only when querying
The headline benefit isn't just cost — it's architectural optionality. You can change query engines next year without moving your data. Your Parquet files on S3 are readable by Athena today, Trino tomorrow, and a custom ML pipeline the day after.

Side-by-Side Comparison

Database vs S3: The Fundamental Differences

Dimension Operational Database S3-based Data Platform
Primary purpose Reliably record what's happening NOW Analyze what has happened, at scale
Unit of work A row (transaction) A file or set of files (Parquet)
Latency Milliseconds for one row Seconds-to-minutes for many rows
Mutability Frequent updates/deletes Mostly append; deletes via rewrite (or lakehouse format)
Storage model Row-oriented, indexed Columnar files in object storage
Schema Strict, defined upfront Often schema-on-read
Storage cost $$$ per TB-month $ per TB-month
Storage & compute Bundled, must scale together Fully separated, scale independently
Scale ceiling Single TB to ~10s of TB Petabytes to exabytes
Concurrency Thousands of concurrent writers OK Designed for few writers, many readers
You use it when Your app needs to save state right now Your team needs to answer questions over history
The two are complementary, not competitive. Operational DBs are the source of truth. S3 is where the analytical copy of that truth accumulates. CDC (Change Data Capture) is the bridge between them.
