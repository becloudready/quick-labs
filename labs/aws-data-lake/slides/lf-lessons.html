The Problem LF Was Built to Solve

Before LF, access to a data lake was pure IAM. IAM is blunt — it controls whether you can touch a service, not what you see when you do.

  • Every analyst needs S3 read on the raw bucket → they see PII columns just by SELECT *
  • Every analyst needs Glue catalog read → they see every table, governed or not
  • Column-level masking? Build it yourself in every query, every view, every dashboard
  • Row-level access? Same — every consumer reimplements the filter independently
What IAM Controls LF Controls
"Can this user use Athena at all?" ✅ —
"Can this user see column volume?" — ✅
"Can this user see rows where month = 12?" — ✅
"Can this user read S3 at all?" Required, but intentionally empty LF issues temp creds at query time
Punchline: IAM stays static per role. LF grants are where access changes. Three users with identical IAM policies can see three different result sets from the same query.

How LF Actually Intercepts a Query

LF doesn't proxy your data. It issues temporary, scoped S3 credentials at query time — Athena reads S3 with those creds, never the user's own IAM.

  1. Athena receives SELECT * FROM oil
  2. Athena → Glue Catalog: "where is this table's data?"
  3. Glue Catalog → Lake Formation: "does this principal have SELECT on this table?"
  4. LF evaluates all grants — named grants, LF-Tag expressions, data cell filters, all combined
  5. LF returns GetDataAccess response — a short-lived IAM credential scoped to: just the allowed columns, just the allowed rows (predicate-pushed), just the registered S3 prefix
  6. Athena uses those temp creds to read S3 — never the user's own IAM
  7. Athena returns rows — already filtered; the user never touches the masked data
The user does NOT need direct S3 read on the data lake bucket  ·  The user CANNOT bypass LF via S3 directly — they have no creds  ·  The audit trail is single-source: every read is a GetDataAccess in CloudTrail

Same Query, Three Views: The Demo Punchline

SELECT * FROM quicklabs_student8_lake.curated_oil LIMIT 5;
Persona Result
Data Lake Admin All 11 columns, all 6,367 rows
Manager (table SELECT grant) All 11 columns, all 6,367 rows
Analyst (column grant — no volume) 10 columns (volume gone), all 6,367 rows
Analyst with data filter (month = 12) 10 columns (volume gone), ~530 rows
Tagged analyst (sensitivity = public) 10 columns (volume tagged restricted), all rows
Five different grant models, five different result sets — all from the same query string.

Governance is declarative (grant a rule) instead of imperative (rewrite the query, build a view, manage S3 prefixes manually).

Audit: Every Read Produces a GetDataAccess Event

Event Payload Contains

  • The IAM principal — who
  • The table being read — what
  • The exact column list LF authorized (after grants + filters + tag rules combined)
  • The Athena query ID — correlate to the SQL itself
  • Timestamp + source IP

What You Can Prove

  • "Show me every query analyst X ran on pii_users in March." → CloudTrail filter on principal + table
  • "Did anyone read column ssn ever?" → filter GetDataAccess where authorized column list contains ssn
  • "Reconstruct what data analyst X could see at any point in time." → cross-reference GetDataAccess + GrantPermissions / RevokePermissions events
Audit capability Traditional IAM + S3 access logs Lake Formation + CloudTrail
Who read a file? Alice read s3://bucket/x.parquet Alice ran query QID-123 on table curated_oil
Which columns were materialized? Unknown — you see the object, not the query Column list in every GetDataAccess event
Was PII column read? Requires Macie or manual inference Direct: filter events by column name
