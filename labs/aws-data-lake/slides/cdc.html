Why CDC: Batch ETL Hits a Wall

The batch pattern runs every N hours: full dump or WHERE updated_at > last_run, land in S3, Glue ETL → Parquet. It works — until it doesn't.

Problem Why It Hurts
Pulls full rows even for tiny changes Wasteful S3 + compute — paying to re-read unchanged data constantly
Dependent on accurate updated_at Tombstone deletes are invisible — deleted rows live forever in your lake
Latency = batch interval Analysts see N-hour-old data — freshness is bounded by schedule, not reality
Source DB hammered during sync Operational impact — batch scans compete with production traffic
CDC fixes all four by reading the database's write-ahead log directly — capturing every change as a delta event (INSERT / UPDATE / DELETE) within seconds, without touching the live table.

How DMS Captures Changes from Postgres

Pipeline — Step by Step

  1. Postgres commits a transaction → WAL record written
  2. DMS replication instance polls via logical replication slot
  3. Postgres emits logical decoding output — column-level before/after values
  4. DMS reformats into change records with operation type (I / U / D)
  5. DMS writes change records to S3 (sub-minute)
  6. Downstream: Glue crawler, Iceberg merge job, or Redshift COPY ingests deltas

Two Postgres Prerequisites

Two RDS Postgres parameters must be set before DMS can read the WAL:

rds.logical_replication = 1
-- sets wal_level=logical after reboot

The DMS user must also hold the rds_replication role — this grants permission to create and consume logical replication slots.

Logical replication slots consume WAL indefinitely if DMS falls behind. Monitor pg_replication_slots to avoid disk exhaustion.

Same SQL, Three Change Files: The Demo Punchline

Three statements against one table produce three distinct S3 files — one per operation:

INSERT INTO crude_oil_daily (trade_ts, open, high, low, close, volume)
       VALUES ('2025-05-12 09:30:00', 72.10, 73.45, 71.88, 73.10, 420000);

UPDATE crude_oil_daily
   SET close = 999.99
 WHERE trade_ts = '2025-05-12 09:30:00';

DELETE FROM crude_oil_daily
 WHERE trade_ts = '2025-05-12 09:30:00';
S3 File Op Contents
LOAD00000001.csv I Full row of the INSERT — all column values at time of write
LOAD00000002.csv U Before-image + after-image of the UPDATE — both states preserved
LOAD00000003.csv D Deleted row's identifying columns — the tombstone row is now visible
Why it matters: Downstream systems can replay exact mutations in order  ·  Deletes are visible for the first time  ·  Idempotent replay with the operation-type column as the discriminator

CDC + Lake Formation = Governed Near-Real-Time Lake

The Flow

  1. DMS writes change records to S3 prefix registered with Lake Formation
  2. Glue streaming job or scheduled crawler merges changes into curated Iceberg / Hudi / Delta table
  3. Curated table inherits same LF grants — column masking, row filters, LF-Tags all apply automatically
  4. Analysts query Athena: fresh-within-seconds data, scoped to what their persona is allowed to see

Operating Model

"Data engineering owns CDC pipelines and tagging. Security owns LF grants and LF-Tag taxonomy. Analysts query Athena. Three teams, zero coordination per new dataset."
Concern Without Lake Formation With Lake Formation
Raw CDC file access Readable by anyone with raw S3 perms Governed — only LF-granted principals can query
PII columns from source Leak into the lake for any SELECT * LF column-level rules strip PII at query time
Region / tenant isolation Requires separate buckets per tenant One bucket, one LF data filter per persona
