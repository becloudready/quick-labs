Encryption — At Rest & In Transit

KMS as the unified encryption control plane across the entire data platform

Encryption at Rest (KMS)

Amazon S3
  • SSE-S3 (AWS managed) or SSE-KMS (your CMK)
  • Bucket default encryption — every new object encrypted automatically
  • Bucket policy can Deny uploads without encryption header
  • CMK gives you key rotation + usage audit in CloudTrail
Amazon Redshift
  • Cluster-level encryption using your KMS CMK
  • Encrypts all data on disk, snapshots, and backups
  • Redshift HSM integration available for regulated workloads
Lake Formation + Glue Catalog
  • Catalog encryption: metadata encrypted with KMS
  • Connection passwords encrypted with KMS
  • Applies to all Glue catalog databases and table definitions

Envelope encryption: KMS encrypts a data key → the data key encrypts the actual data. You never send raw data to KMS. The CMK never leaves KMS hardware.

Encryption in Transit (TLS)

S3 — Force HTTPS
  • Bucket policy with Deny on aws:SecureTransport: false
  • Blocks any HTTP (unencrypted) request to the bucket
  • Apply to both source and destination buckets
Redshift
  • Require SSL in parameter group: require_ssl = true
  • TLS 1.2 minimum for all client connections
  • JDBC/ODBC drivers enforce certificate validation
Lake Formation / Athena / OpenSearch
  • All API calls to AWS services are HTTPS (enforced by AWS)
  • Athena query results written to S3 — inherit bucket encryption
  • OpenSearch: enforce HTTPS + TLS security policy on domain
Secrets Manager
  • All credentials (DB passwords, API keys) stored encrypted
  • Never inline credentials in IAM policies or code
  • Automatic rotation reduces long-lived secret risk
Key Takeaway Use one CMK per data classification level (raw, enriched, consolidated) — not one key for everything. Separate keys give you independent rotation schedules, access policies, and audit trails.

Network-Level Data Isolation

S3 bucket policies · VPC endpoints · private network paths — keep data traffic off the public internet

🪣

S3 Bucket Policies

  • Restrict access to specific VPC endpoints only (aws:sourceVpce)
  • Deny all requests not originating from approved VPC
  • Restrict to specific IAM principals (service roles, not individuals)
  • Enforce HTTPS — deny aws:SecureTransport: false
  • Block public ACLs at the account level (S3 Block Public Access)
  • Scope resource to prefix level, not whole bucket where possible
Deny: "aws:sourceVpce" ≠ your VPC endpoint ID
🔌

VPC Endpoints

  • Gateway endpoint — S3 & DynamoDB: free, route-table-based, no ENI
  • Interface endpoint — all other services: creates ENI in your subnet, private IP, SG controls
  • Once endpoint is in place, traffic never traverses the public internet
  • Endpoint policy limits which S3 buckets the endpoint can reach
  • Use separate endpoints per subnet for HA across AZs
Services: S3, STS, Glue, Athena, Secrets Manager, KMS
🏗️

VPC Architecture for Data Platform

  • All compute (Glue, Lambda, DMS, Redshift) in private subnets — no internet gateway route
  • Security groups: allow only required service-to-service ports
  • NACLs as a coarse subnet-level backstop
  • No public IP addresses on data service instances
  • NAT gateway only if external dependencies genuinely require outbound internet
  • Separate VPCs per environment (dev / staging / prod) with Transit Gateway or peering for approved paths
Rule: if data can leave via public IP, the network design is wrong
Private Data Path (no internet) Lambda (private subnet) → VPC Endpoint (S3 Gateway) → S3 Bucket (VPC-only policy) → KMS (Interface endpoint) Zero public internet traversal
Key Takeaway Bucket policy + VPC endpoint + SG = defense in depth at the network layer. A bucket policy alone can be bypassed by an over-privileged role elsewhere; combining it with VPC enforcement closes that gap.

IAM + Lake Formation Hybrid Models & Org-Level Governance

How IAM and Lake Formation permissions combine · SCPs and permission boundaries in production

Permission Evaluation Order

1
SCP (Service Control Policy)
Org-level ceiling. No principal in the account can exceed what the SCP allows, regardless of their IAM policy. Explicit Deny here is final.
2
Permission Boundaries
Role-level ceiling. The effective permission is the intersection of the identity policy and the boundary. Used to safely delegate role creation to teams.
3
IAM Identity Policy
Must explicitly Allow the service API (e.g. s3:GetObject, glue:GetTable). No implicit Allow — if not listed, denied.
4
Lake Formation Data Permission
For Athena / Spectrum queries: principal must have BOTH IAM Allow AND Lake Formation grant. Either one absent = access denied. This is the hybrid model.
5
Resource Policy (bucket policy, KMS key policy)
Evaluated at the resource. Can further restrict even if steps 1–4 allow. An explicit Deny here overrides everything above.

SCPs — Org-Level Controls

Common Data Platform SCPs

  • Deny disabling S3 Block Public Access
  • Deny deleting CloudTrail trails
  • Deny creating unencrypted S3 buckets
  • Restrict to approved regions only
  • Deny IAM actions that could create admin roles without boundaries

Permission Boundaries in Practice

  • Attach a boundary to every role a team creates
  • Boundary says: "this role can never exceed read-only on S3"
  • Team can grant the role any identity policy — boundary caps it
  • Prevents privilege escalation through role creation
Explicit Deny Rule An explicit Deny at ANY layer (SCP, boundary, IAM, resource policy, LF) overrides every Allow above it. When debugging access denials, check all five layers — not just IAM.

IAM + LF Hybrid: the critical rule

  • IAM controls which AWS APIs the principal can call
  • Lake Formation controls which data the query can see
  • Both must Allow — missing either blocks access
  • LF column/row filters apply after IAM clears the API call
Key Takeaway Never treat IAM + Lake Formation as OR. Access requires both to Allow. SCPs set the org ceiling; permission boundaries prevent escalation; LF adds data-level granularity IAM cannot provide.

Threat Detection & Audit — GuardDuty + CloudTrail

Detect threats in real time · maintain an immutable record of every data access

GuardDuty — Continuous Threat Detection

🚨
Unusual API Activity Flags calls to S3, Athena, or Glue from unfamiliar IPs, geolocations, or at unusual hours. Baseline is learned automatically over 2 weeks.
📤
S3 Data Exfiltration Signals Detects high-volume GetObject calls, large data downloads, or objects being copied to buckets in other accounts or public.
🔓
Compromised Credentials IAM role being used from a Tor exit node, known malicious IP, or from a region the role has never used before.
⚙️
Data Platform Specific Findings Athena queries scanning unexpectedly large volumes; Glue crawler accessing buckets outside its configured scope; Lambda accessing S3 cross-account.
Response flow GuardDuty Finding → EventBridge Rule → SNS Alert → Lambda Remediation

CloudTrail — Deep Dive Audit

📋
What CloudTrail Records Every API call: who (principal ARN), what (API action), when (timestamp), where (source IP), on what (resource ARN) — including read/write events on S3, Glue, Athena, Lake Formation, Redshift.
🏷️
Data Events (critical for data platforms) S3 object-level events (GetObject, PutObject, DeleteObject) are not logged by default — enable S3 data events on sensitive buckets explicitly.
🔍
Detecting Anomalies with Athena Query CloudTrail logs in S3 with Athena. Find: who accessed a sensitive table in the last 7 days; which role made the most AssumeRole calls; denied access patterns.
🔒
Protecting the Audit Trail Write CloudTrail logs to a dedicated S3 bucket with: Object Lock (WORM), no delete permission for any role, CloudWatch alarm on DeleteTrail or StopLogging API calls.
Audit query CloudTrail → S3 → Athena table → SQL query → Access report
Key Takeaway GuardDuty detects what is happening now; CloudTrail tells you what happened and who did it. Both are required for a complete security posture — detection without audit history makes forensics impossible.

Real-World Scenario — Least Privilege Across the Full Pipeline

Every service, every stage, every layer — security controls applied end-to-end

🪣 S3
Raw Layer
IAM
Ingest role has s3:PutObject on raw prefix only. No GetObject, no DeleteObject.
Encryption
SSE-KMS with raw-data CMK. Bucket policy denies non-encrypted uploads (s3:x-amz-server-side-encryption).
Network
Bucket policy denies any request not from the designated VPC endpoint. Block Public Access enabled.
🏛️ Lake Formation
Enriched Layer
IAM
Analyst role has IAM Allow on glue:GetTable and athena:StartQueryExecution only.
Encryption
Glue catalog encrypted with enriched CMK. Column data filtered by LF before reaching Athena — no raw PII exposed.
Network
Lake Formation + Glue API calls via Interface VPC endpoints. No public Glue service endpoints in use.
🔴 Redshift
Consolidated
IAM
Redshift cluster role scoped to specific S3 prefixes for COPY. Spectrum queries checked by LF — table-level + column-level grants.
Encryption
Cluster encrypted with KMS CMK at creation. Automated snapshots inherit encryption. require_ssl=true in parameter group.
Network
Cluster in private subnet. Security group allows port 5439 from BI tool subnet only. No public endpoint.
🔍 OpenSearch
Search Layer
IAM
FGAC: reader role mapped to a backend role that grants read-only on specific indices only. No DELETE index permission.
Encryption
Domain encryption at rest with KMS. Node-to-node encryption enabled. HTTPS enforced with TLS 1.2+ policy.
Network
Domain in VPC — no public access. Security group permits inbound 443 from application subnet only.
Across all stages: CloudTrail logs every API call · GuardDuty monitors for anomalies · SCP prevents disabling encryption or logging · CMKs are per-layer with independent rotation · No human has direct production data access — only service roles.
Module 11 Summary Security = IAM (who) + Lake Formation (what data) + KMS (encrypted at rest) + VPC endpoints (no public path) + GuardDuty (detect threats) + CloudTrail (prove it). All six layers, every stage.
