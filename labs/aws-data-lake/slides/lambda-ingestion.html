Event-Driven Ingestion — S3 as the Landing Zone

Three Use Cases, One Landing Zone

S3 acts as the durable landing zone for all ingestion patterns. S3 ObjectCreated events are the trigger — the rest of the architecture differs per use case.

UC-1 — SQS Fan-In (Async)

→Field devices PUT .jpg + sidecar → S3 raw (images/)
→S3 event → SQS image-events queue
→SQS → Lambda image_metadata_handler
→Lambda → S3 curated (image-metadata/)
→Poison messages → DLQ after maxReceiveCount=5

UC-2 — S3 Direct (Batch Drop)

→Batch scheduler PUTs .csv → S3 raw (drop/)
→S3 event → Lambda batch_file_handler (async invoke)
→Lambda → S3 curated (batch/yyyy/mm/dd/)
→Async failures after 2 retries → Lambda async DLQ

UC-3 — Lambda-as-ETL

→Partner SFTP PUTs .csv → S3 raw (oil_drop/)
→S3 event → Lambda csv_to_parquet_curated
→Lambda → S3 curated (Hive partition path)
→Lambda → Glue Catalog CreatePartition
→Lambda → Redshift COPY via redshift-data API
Cross-cutting: CloudWatch Logs + Metrics + Alarms for all Lambdas  ·  CloudTrail data events for all S3 buckets

Event-Driven Ingestion — S3 as the Landing Zone

UC-1: SQS-Fronted Fan-In Sequence

  1. Device → S3 raw: PUT images/tank-1.jpg + .json sidecar
  2. S3 raw → SQS: ObjectCreated event delivered to queue
  3. SQS → Lambda: poll batch (size 10, window 5s)
  4. Lambda → S3 raw: GetObject head + sidecar read
  5. Lambda → S3 curated: PutObject image-metadata/...json
  6. Lambda → SQS: success → DeleteMessage
  7. On repeated failure (maxReceiveCount=5) → message moved to DLQ by SQS automatically

Why SQS in Front?

  • Decouples producer spike from Lambda concurrency limits — queue absorbs bursts
  • SQS DLQ catches poison messages (malformed files, permission errors) without retrying forever
  • Batch window allows Lambda to process 10 events per invocation → cheaper per-message cost
Config Value Why
Batch size 10 Cost efficiency per invocation
Batch window 5s Small latency tradeoff for batching
maxReceiveCount 5 Retry 5× before DLQ quarantine
Visibility timeout > Lambda timeout Prevents duplicate processing

Event-Driven Ingestion — S3 as the Landing Zone

UC-3: Lambda-as-ETL Sequence (The Complex One)

  1. Partner → S3 raw: PUT oil_drop/oil-2025-05-12.csv
  2. S3 raw → Lambda: async invoke (ObjectCreated)
  3. Lambda → S3 raw: GetObject (stream CSV)
  4. Lambda: validate schema, transform → Parquet, group by trade_date
  5. Lambda → S3 curated: PutObject oil_curated/year=.../month=.../day=.../...parquet
  6. Lambda → Glue Catalog: CreatePartition (Hive-style values)
  7. Lambda → Redshift: redshift-data ExecuteStatement COPY ... FORMAT PARQUET
  8. Lambda → S3 raw: PutObjectTagging ingest-status=processed

Key Design Choices

  • Hive partition path written by Lambda — Glue Catalog discovery is immediate, no crawler needed after each drop
  • Redshift COPY uses redshift-data API (async) — Lambda doesn't hold open a DB connection; Redshift polls on its own
  • PutObjectTagging at end = idempotency marker — re-triggered Lambda events skip already-tagged files
Partition path convention: 
oil_curated/
  year=2025/
    month=05/
      day=12/
        part-0001.parquet
Athena and Glue recognize this pattern natively — no manual schema repair needed.
