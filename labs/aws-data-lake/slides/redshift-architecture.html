Module 8 — Amazon Redshift

Amazon Redshift Architecture & Optimization

Redshift takes columnar-storage physics (column blocks, type-specific compression, zone maps) and adds three things you can't build yourself:

1. Distributed Architecture

One query, many machines, results stitched together automatically by the leader node.

2. Managed Control Plane

Backups, replication, scaling, security — all handled. You write SQL; AWS runs the infrastructure.

3. Columnar-Aware Optimizer

Query plans designed to maximize column pruning and zone-map block skipping.

What this module covers

  • How a query becomes work distributed across many machines (slides 2-4)
  • How to physically lay out data so queries are fast (slides 5-6)
  • How to share a cluster fairly across many users (slide 7)
  • How to diagnose and fix slow queries (slide 8)
  • How to maintain data freshness (slide 9)
  • How to pick a deployment model and protect the data (slides 10-11)

Architecture

Architecture: Leader + Compute Nodes

Leader Node (1 per cluster)

  • Receives the SQL, parses, plans, and optimizes the query
  • Splits the plan into pieces and ships them to compute nodes
  • Aggregates partial results returned by compute nodes
  • Returns final result set to your client
  • Holds NO table data — it's purely the orchestrator

Compute Nodes (1 to 128+ per cluster)

  • Each holds a slice of the cluster's data
  • Each runs the piece of the query plan against its own data
  • Returns its partial result back to the leader
  • The unit you scale for more storage or query throughput

Why this beats a single-server database

Query: SELECT region, SUM(total) FROM orders GROUP BY region over 100 billion rows

Single-server DB:

One machine reads all 100B rows serially

Redshift with 10 compute nodes:

Each reads 10B rows in parallel → computes partial sum per region → sends partials to leader → leader merges → ~10× faster (assuming even distribution)

The parallelism factor is determined by the number of slices across all compute nodes — not just the node count. A 10-node cluster with 4 slices/node gives 40-way parallelism.

Architecture

Slices: The Node's Disk Storage, Partitioned for Parallelism

A slice is a partition of the compute node's disk storage. Each row of every table lives on exactly one slice. Slices can scan, filter, and aggregate their own data without coordinating with any other slice.

Slices per node (examples)

Node type Default slices
dc2.large 2
dc2.8xlarge 16
ra3.large / ra3.xlplus 2
ra3.4xlarge 4
ra3.16xlarge 16

Why slices matter

  • A table's rows are distributed across slices (not nodes). A 10-node ra3.4xlarge cluster has 40 slices — that's the parallelism factor.
  • If one slice owns more rows than others, it becomes the bottleneck. The query is only as fast as its slowest slice.
  • This is data skew — the most common reason a Redshift query is slow.

Inspecting skew

SELECT slice, COUNT(*) AS rows
FROM stv_tbl_perm
WHERE name = 'orders'
GROUP BY slice
ORDER BY rows DESC;
If the top slice has 10× more rows than the bottom, you have significant skew. The fix is almost always choosing a better DISTKEY.

Architecture

Distribution Styles: KEY, ALL, EVEN, AUTO

How you distribute rows across slices is the single biggest performance decision in Redshift.

KEY — rows hashed by a column

CREATE TABLE orders (...) DISTKEY (customer_id);

All rows with same customer_id land on same slice. Best when: frequently joining on that column. Worst when: column has skewed cardinality.

ALL — every node holds a full copy

CREATE TABLE regions (...) DISTSTYLE ALL;

Full table replicated to every compute node. Best when: small table (< few million rows), joined to many big tables. Worst when: large table.

EVEN — round-robin across slices

CREATE TABLE events (...) DISTSTYLE EVEN;

Rows distributed round-robin. Best when: table isn't joined to other big tables. No skew risk. Worst when: frequently joining on a specific column.

AUTO — Redshift decides

Picks ALL for small tables, EVEN for medium, KEY when it detects a clear pattern. Re-evaluates as the table grows. Good default when you're unsure.

Decision tree: "Does this table get joined to a big table on a specific column?"
→ Yes: that column is your DISTKEY
→ No, small + read-heavy: DISTSTYLE ALL
→ No, no clear join column: EVEN
→ Don't know yet: AUTO

Data Layout

Sort Keys: Compound vs Interleaved

Sort keys decide the physical row order within each slice. That ordering enables zone-map block pruning — Redshift can skip entire 1MB blocks without reading them if the block's min/max values don't match a WHERE predicate.

Compound sort key

CREATE TABLE orders (...)
  SORTKEY (order_date, region);
  • Sorts hierarchically — first by order_date, then by region within each date
  • The first column is dominant — pruning works great if WHERE includes order_date
  • Weak pruning if you filter only on region without order_date
  • Best for: queries that almost always filter on the first column

Interleaved sort key

CREATE TABLE orders (...)
  INTERLEAVED SORTKEY (order_date, region, customer_id);
  • Gives equal weight to all sort key columns
  • Pruning works well whether you filter on order_date, region, OR customer_id
  • Data arranged in a multi-dimensional Z-order curve
  • Best for: ad-hoc analytical workloads with unpredictable filters
  • Trade-off: slower to maintain — VACUUM REINDEX required periodically
Rule of thumb: If 80% of queries filter on the same first column (usually a date), use compound. If queries filter on different columns roughly equally, use interleaved — but accept the maintenance cost of periodic VACUUM REINDEX.

Data Layout

Compression Encodings

Encoding Best for Ratio
AZ64 Numeric, dates, timestamps (Amazon's default) 2-5×
ZSTD General-purpose fallback, strings 3-10×
BYTEDICT Strings with low cardinality (< 256 distinct values) 10-40×
DELTA / DELTA32K Sequential integers, dates 10-30×
MOSTLY8/16/32 Integers that fit in smaller types most of the time 2-4×
RUNLENGTH Columns with long runs of the same value 10-100×
TEXT255 / TEXT32K VARCHAR with short distinct values 5-10×
RAW No compression (only if column is uncompressible) 1×

Don't pick by hand for first iteration

Let Redshift sample your data during COPY and apply encodings automatically:

COPY orders FROM 's3://bucket/path/'
IAM_ROLE 'arn:aws:iam::ACCT:role/RedshiftCOPYRole'
COMPUPDATE ON;

Or preview without applying:

ANALYZE COMPRESSION orders;
Why compression matters beyond storage: Smaller columns = fewer disk blocks read per query = faster scans. BYTEDICT on a low-cardinality string column can make a GROUP BY on that column 5-10× faster even ignoring storage savings.

Resource Management

Workload Management (WLM) + Concurrency Scaling

WLM keeps one user's slow report from starving everyone else's dashboards.

Auto WLM (default since 2019)

You define query queues (2-8 typically), assign queries by user/group/query group. Redshift dynamically tunes memory + concurrency based on actual workload.

SET query_group TO 'reports';
SELECT ... ;  -- runs in "reports" queue
RESET query_group;

Each queue has independent memory and concurrency budgets. A misbehaving heavy query in analytics can't drag down queries in dashboards.

Concurrency Scaling

When a queue is full and queries start waiting, Redshift transparently spins up additional shadow clusters. You pay only for the seconds those shadow clusters actually run. Each main cluster gets 1 hour of free concurrency scaling credit per 24 hours of operation.

Short Query Acceleration (SQA)

Identifies short queries (sub-second) and routes them to a fast lane so they aren't stuck behind a 10-minute aggregation. On by default.

Queue design tip: Create at least three queues: dashboards (high memory, low concurrency), reports (medium), etl (low memory, runs off-hours). Prevents ETL from blocking dashboards.

Tuning

Query Performance Tuning: Diagnose Then Fix

Step 1 — Read the execution plan

EXPLAIN
SELECT region, SUM(total)
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE order_date >= '2024-01-01'
GROUP BY region;
DS_ Line Meaning
DS_DIST_NONE Both sides co-located on same slice ✅ Best
DS_DIST_INNER Inner table shuffled across nodes ⚠️ Costly
DS_DIST_BOTH Both tables shuffled ❌ Very costly
DS_BCAST_INNER Inner table broadcast to every slice ⚠️ OK for small tables

Step 2 — Check actual execution

SELECT query, elapsed, rows, bytes
FROM stl_query
WHERE userid = 100
ORDER BY starttime DESC
LIMIT 20;

Step 3 — Common pitfalls

Symptom Likely cause Fix
One slice does 80% of work Skewed DISTKEY values Pick higher-cardinality DISTKEY or EVEN
Full-table scan on date column No sort key on that column Add SORTKEY (date_column)
Slow aggregate on wide table No column-level encoding ANALYZE COMPRESSION and recreate
Joins show DS_DIST_BOTH DISTKEY mismatch between joined tables Match DISTKEYs across joined tables
New columns added, queries got slower Statistics out of date Run ANALYZE
Most slow Redshift queries are slow for one of three reasons: wrong distribution, wrong sort key, or stale statistics. Check in that order.

Maintenance

Maintenance Ops: VACUUM and ANALYZE

VACUUM — reclaim space and re-sort

DELETE and UPDATE don't immediately remove rows — they mark them as "ghosted." VACUUM reclaims the space and re-sorts new INSERTs back into sort key order.

VACUUM orders;              -- full vacuum (sort + reclaim)
VACUUM DELETE ONLY orders;  -- just reclaim space
VACUUM SORT ONLY orders;    -- just re-sort
VACUUM REINDEX orders;      -- for interleaved sort keys

ANALYZE — refresh statistics for the query planner

Without fresh stats, the planner picks bad join orders, wrong distribution choices, sub-optimal scan paths.

ANALYZE orders;                      -- refresh all column stats
ANALYZE orders (order_date, region); -- just those columns

When to run which

Trigger Run
After large COPY / INSERT ANALYZE (auto handles most cases)
After bulk DELETE / UPDATE VACUUM DELETE ONLY + ANALYZE
Weekly maintenance window VACUUM (full)
Interleaved sort key, every 1-2 months VACUUM REINDEX
Queries getting slower for no reason ANALYZE, then VACUUM
Redshift auto-manages ANALYZE and VACUUM for most workloads since 2020. You need to intervene manually mainly when: you're doing very large one-time loads, running bulk deletes, or you have interleaved sort keys that need REINDEX.

Deployment

Redshift Serverless vs Provisioned

Provisioned Clusters

You pick the node type and count, pay per node-hour 24/7. Best when: predictable steady workload, you can amortize always-on cost, need fine-grained WLM control.

Redshift Serverless

No nodes to manage. Pick base RPU capacity — Redshift scales compute up/down automatically. Pay per RPU-second only while queries are running. Best when: spiky workload, cluster sits idle for long stretches, want zero infrastructure management.

Pattern Pick
Constant high load 24/7 Provisioned — cheaper per-second at saturation
Batch jobs 2h/day, idle 22h Serverless — no idle cost
Unpredictable spikes (analyst BI) Serverless — pays only for the spike
Need specific WLM config / node type Provisioned — full control
First-time POC or evaluation Serverless — zero setup
Hybrid pattern: Provisioned for steady-state ETL workload + serverless workgroups for ad-hoc analyst queries, sharing the same data via data sharing. You get cost efficiency on both ends.

Operations

Backup Strategies

Automated snapshots (on by default)

  • Taken every 8 hours OR after every 5 GB of changes, whichever comes first
  • Stored in S3 (Redshift-managed, doesn't count against your bucket quota)
  • Default retention: 1 day (configurable 0 to 35 days)

Manual snapshots

  • Triggered on-demand; retained until you explicitly delete them
  • Survive cluster deletion
  • Use for: pre-upgrade checkpoints, end-of-quarter freezes

Cross-region snapshot copy

Automatically copy snapshots to a secondary AWS region for DR.

Restore process

aws redshift restore-from-cluster-snapshot \
  --cluster-identifier restored-prod \
  --snapshot-identifier rs:prod-2024-12-25-23-00-00

Restores create a NEW cluster — apps need to be re-pointed to the new endpoint.

Minimum-viable backup policy

Tier Config
Dev / sandbox Default automated (1-day retention), no cross-region
Staging 7-day retention, manual snapshot before major changes
Production 35-day retention, cross-region copy enabled, quarterly manual snapshots retained indefinitely
RPO / RTO estimates: With 35-day automated snapshots + cross-region copy, your RPO is ≤ 8 hours (last snapshot), RTO is 15-30 minutes (Redshift restore time for a small cluster). For tighter RPO, use Redshift data sharing or a real-time CDC pipeline as the analytical replica.
