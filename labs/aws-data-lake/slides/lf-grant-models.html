Lake Formation — Named vs Tag-Based Grant Models

Two Ways to Grant in Lake Formation

When you click Grant in the LF console, you choose a model. Same button — two fundamentally different mental models.

Named Data Catalog Resources

👆 Point at specific databases / tables / columns by exact name

You select exactly what you want to share. If a table doesn't exist yet, it isn't covered — every new resource needs a new grant.

SQL analogue
GRANT SELECT ON quicklabs.curated_oil
  TO student7;

Resources Matched by LF-Tags

🧲 Define a tag expression — LF matches all matching resources at query time

You write a predicate, not a resource list. New tables tagged correctly are automatically covered — zero new grants required.

SQL analogue
GRANT SELECT ON ANY TABLE
  WHERE classification = 'public'
  TO student7;

Named Data Catalog Resources

You pick specific things by name, optionally down to individual columns:

  • Pick database = quicklabs_student8_lake
  • Pick table = curated_oil
  • Optionally pick specific columns — e.g. all except volume
Property Behavior
Scope Static — exactly the resources you ticked at grant time
New tables created tomorrow Not covered — requires a new grant per resource
Audit / listing Easy: Data Permissions page shows db → table directly
Best for Small data lakes, one-off grants, ad-hoc sharing
SQL analogue
GRANT SELECT ON quicklabs_student8_lake.curated_oil TO student7;
Named grants work well for small lakes with stable schemas. They couple your security model to your data catalog — every new table is a support ticket to security.

Resources Matched by LF-Tags

You write a tag expression; LF resolves it at runtime against the current tag state of every resource:

Tag expression example
sensitivity = public AND department = finance

Applies to every resource carrying these tags — today and in the future.

Property Behavior
Scope Dynamic — defined by predicate, not a resource name
New tables tagged accordingly tomorrow Automatically covered — zero new grants required
Audit / listing Harder — must join tag expression with current tag attachments across resources
Best for Large orgs, many tables, consistent classification policy
SQL analogue
GRANT SELECT ON ANY TABLE WHERE classification = 'public' TO student7;

When the Difference Matters

Scenario Named Grants LF-Tag Grants
1 table, 1 consumer 1 grant 1 grant + 1 tag attachment
50 tables, 1 consumer, same access 50 grants 1 grant + 50 tag attachments
Add a 51st table next week +1 more grant — manual action required Just tag the new table — grant auto-applies
Audit "what can student7 read?" Look at Data Permissions list — direct Cross-reference tag expression + every table's current tags
Coordination cost Every new table = ticket to security team Data eng tags; security wrote the rule once
The punchline: Named grants couple access control to resource creation — security acts every time data engineering ships a new table. Tag-based grants decouple them. In a real organization, this is the difference between "security is a bottleneck" and "security scales."
