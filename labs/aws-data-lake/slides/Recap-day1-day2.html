Module 5 — Lake Formation: Governance Foundations

   Central access control layer that sits on top of S3 and IAM
What Lake Formation Is (and Isn't)
  • Data Catalog — central metadata store for all S3 datasets
  • NOT a Glue ETL replacement — governs access, not transformation
  • Works on top of S3 — LF registers S3 paths, then takes over from raw IAM
  • Principals (users/roles) query via Athena, Redshift Spectrum, or EMR
  • LF acts as the enforcement point — Athena asks LF if you can see the table
  • Glue crawler still populates catalog schema; LF controls who sees it
  • Service-linked role: AWSServiceRoleForLakeFormationDataAccess is created automatically
Key mental model: S3 holds the bytes. Glue catalogs the schema. Lake Formation decides who can read it.
Setup Flow & Dual-Layer Model
  • Step 1: Designate a Data Lake Administrator in the LF console
  • Step 2: Register S3 locations — LF now manages access to those paths
  • Step 3: Create databases & tables in the Data Catalog (Glue crawler or manual DDL)
  • Step 4: Grant LF permissions to IAM principals (users, roles, groups)
  • Step 5: Grant Athena workgroup or Redshift namespace access to query
  • Revoke the default "IAMAllowedPrincipals" super-grant — otherwise IAM alone can bypass LF
IAM + Lake Formation dual-layer — both must Allow:
IAM = can the principal call the Athena / Glue API?
LF = can the principal see this database / table / column?
A single Deny in either layer blocks the query.
Access Path Athena query → Lake Formation check → IAM check → S3 data read Both Lake Formation AND IAM must allow — a single deny blocks the query

Module 5 — Fine-Grained Access Control & LF-Tags (ABAC)

   Column · row · cell-level filters + attribute-based access that scales without per-table grants
Permission Hierarchy
🗄️ Database-level Access to all tables in a database
📋 Table-level Access to one specific table
📊 Column-level Include or exclude specific columns per role
🔎 Row-level filter WHERE-clause condition attached to the principal
🔐 Cell-level security Column exclusion + row filter combined
Example: analyst role sees all rows but no SSN column. Contractor role sees only their region's rows and no salary columns.
LF-Tags for ABAC (Attribute-Based Access Control)
  • Assign tags to resources (databases, tables, columns)
  • Assign matching tags to principals (IAM users / roles)
  • Access = tag match — no explicit per-table grants needed
  • New tables tagged correctly are automatically accessible to matching principals
  • Eliminates the O(tables × roles) grant explosion of traditional RBAC

  • Cross-account sharing: share via Resource Links in the target account; analysts query without copying data
  • CloudTrail logs every LF access decision — full audit trail for compliance
ITAR scenario: tag tables classification=itar, grant only cleared roles the matching tag — one grant covers every future ITAR table automatically.
LF-Tag Flow Principal (role) has tag → LF evaluates tag match → Query on filtered view (column + row) Audit: every access decision recorded in CloudTrail · ITAR / compliance ready

Module 7 — Change Data Capture with AWS DMS

   Stream row-level changes from operational databases into the warehouse — no nightly full reloads
DMS Pipeline & Task Modes
Source
🐘 PostgreSQL / RDS
Logical replication slot enabled
→
DMS Replication
⚙️ Replication Instance
Reads WAL / redo logs
→
Target
🔴 Redshift / S3
Rows applied in near real-time
  • full-load: initial snapshot of all existing rows
  • cdc: capture ongoing inserts / updates / deletes only
  • full-load-and-cdc: snapshot first, seamlessly switch to live capture
  • Source prerequisite: logical replication ON + replication slot created
  • Replication Instance size drives throughput — scale up for large tables
  • LOB columns (BLOB / CLOB): use Limited LOB Mode for best performance
  • Batch apply mode: buffer changes, write in batches to Redshift — better throughput than row-by-row
Production Considerations
  • Schema changes: DMS does NOT auto-alter the target — apply DDL manually or use a schema-change event handler
  • Replication lag: monitor CDCLatencySource & CDCLatencyTarget in CloudWatch
  • Error handling: configure task-level error policies (stop / continue / log) per table; set CloudWatch alarms
  • Data Catalog consistency: new source columns won't appear in Glue until manually updated
  • Primary key requirement: tables without PKs can use full-load; CDC requires a PK or unique index for correct row tracking
  • CDC vs Batch ETL: CDC wins on latency (<1 min lag) and source load; batch wins on simplicity — for a live DW feeding dashboards, CDC is the right choice
  • Enterprise use case: keep Redshift DW synchronized with PostgreSQL operational DB without nightly full ETL jobs
Teardown check: after the training, delete the DMS Replication Instance — it bills by the hour even when the task is stopped.
