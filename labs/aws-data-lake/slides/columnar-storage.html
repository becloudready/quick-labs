Columnar Storage — Why Parquet, Redshift, and BigQuery are Fast

The One Insight: Row vs Columnar Layout on Disk

Row Storage (Postgres, MySQL, RDS)

Block 1:
[r1.col1|r1.col2|r1.col3|r1.col4]
[r2.col1|r2.col2|r2.col3|r2.col4]
...

Each disk block holds a few full rows.
Optimized for: "fetch one whole row fast."

Columnar Storage (Redshift, Parquet, BigQuery)

Block A: [r1.col1, r2.col1, r3.col1, r4.col1 ...]
Block B: [r1.col2, r2.col2, r3.col2, r4.col2 ...]
Block C: [r1.col3, r2.col3, r3.col3, r4.col3 ...]

Each disk block holds values of one column across many rows.
Optimized for: "scan a few columns across millions of rows fast."

Query pattern Row storage Columnar storage
Fetch one full row ✅ Single I/O ❌ Reads every column block
Scan 3 columns × 1B rows ❌ Reads every row ✅ Reads 3 of N column blocks
Analytical queries are 99% the second pattern. That's why every analytics system is columnar.

Why Columnar Wins for Analytics: The I/O Argument

SELECT region, SUM(total_usd) FROM orders
 WHERE order_date >= '2024-01-01'
 GROUP BY region;

This needs 3 columns (region, total_usd, order_date) out of a typical 50-column orders table.

Storage What's Actually Read Off Disk
Row storage All 50 columns of every row — even the 47 you don't need
Columnar storage Only the 3 column blocks — the other 47 are skipped entirely

The Math at Scale

~1.7×
5-column table
3 of 5 read (60%)
~17×
50-column table
3 of 50 read (6%)
~170×
500-column table
3 of 500 read (0.6%)
The wider the table, the bigger the win. Modern analytics tables are wide — clickstream events, sensor readings, ML feature stores often have 100s to 1000s of columns.

Why Columnar Wins: The Compression Argument

In row storage, a block holds mixed types — compression does OK. In columnar, each block holds one type, one logical meaning — compression exploits patterns that can't exist in mixed blocks.

Block (region): ["US-W", "US-E", "US-W", "EU", "US-E", "US-W", ...]
Column Shape Redshift Encoding What It Does Typical Ratio
Few distinct values (region: US-W / US-E / EU) BYTEDICT Map each distinct value to a small integer ID ~40×
Sequential dates / IDs (order_id: 1001, 1002...) DELTA Store each value as offset from previous ~30×
Long runs of same value (status: "active"...) RUNLENGTH Store "value × N consecutive copies" 10–100×
Numerics with limited range AZ64 Specialized numeric compression 2–5×
General-purpose fallback ZSTD Modern general-purpose compression 3–10×
A typical Redshift table occupies 3–10× less disk than the same data in Postgres — which also means 3–10× less I/O per query.

Redshift's Bonus Tricks: Zone Maps + Sort Keys

Zone Maps

Block-level min/max metadata. Before reading a block, the planner asks: "could this block contain rows matching my WHERE clause?"

WHERE order_date BETWEEN '2024-01-15'
                     AND '2024-01-17'
Block min(order_date) max(order_date) Read?
Block 1 2020-01-01 2020-12-31 skip
Block 2 2023-01-01 2023-12-31 skip
Block 3 2024-01-10 2024-01-20 read
Block 4 2024-02-01 2024-02-28 skip

Sort Keys

Physically cluster related rows so zone maps become highly selective:

CREATE TABLE orders (...)
  SORTKEY (order_date);

Now order_date values are tightly clustered — block 1 holds Jan 2020, block 2 Feb 2020, and so on. Zone maps skip entire months in a single check.

With years of data, queries often scan fractions of 1% of the table due to zone map skipping.
The lesson: Columnar storage gives column-level pruning. Zone maps + sort keys give row-level pruning within those columns. They compound.

When Row Storage Still Wins

Workload Row Storage (Postgres) Columnar (Redshift)
Fetch one full row by ID ✅ Single I/O, ms latency ❌ Reads every column block
Update one row ✅ Modify one block ❌ Touches many column blocks
Insert one row ✅ Append to one block ❌ Splits across many column blocks
Thousands of concurrent transactions ✅ Designed for it ❌ Throughput bottleneck
SELECT region, SUM(total) GROUP BY region over 1B rows ❌ Reads every row ✅ Reads 1 column
Aggregations, joins on massive history ❌ Too slow ✅ Native workload
Compression of cold archival data ❌ Mixed-type blocks ✅ 3–10× smaller

Practical Pattern in Modern Data Platforms

1 Operational database (Postgres / RDS / Aurora) — row-oriented, OLTP, source of truth for applications
2 CDC / streaming (AWS DMS, Kinesis, Kafka) — bridges operational changes to the analytics layer
3 Data lake / warehouse (Parquet on S3 + Athena, or Redshift) — columnar, OLAP, queryable history at any scale
