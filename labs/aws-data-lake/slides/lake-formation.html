Section 1 of 3 — Introduction

AWS Lake Formation

Introduction, Grant Models & Demo Lessons

A service for centrally governing, securing, and globally sharing data used for analytics and ML.

  • Manages fine-grained access control on data lake data in S3 and metadata in the AWS Glue Data Catalog
  • Provides its own permissions model that augments IAM — doesn't replace it
  • Grant/revoke mechanism similar to a relational database (RDBMS)
  • Permissions enforced at granular levels: column, row, and cell
  • Consistent enforcement across AWS analytics and ML services
  • Supports sharing data across multiple AWS accounts, AWS Organizations, or directly with IAM principals

Permissions apply to

  • IAM users and roles
  • Users / groups federated through an external IdP

Fine-grained controls cover

  • Database level
  • Table level
  • Column level
  • Row level (data filters)
  • Cell level (row + column combined)

Where Lake Formation Enforces Permissions

Permissions apply to IAM users and roles, and to users/groups federated through an external IdP. Enforcement is consistent across all integrated AWS analytics and ML services.

Integrated AWS services

AWS Service LF Permissions Enforced
Amazon Athena ✅
Amazon QuickSight ✅
Amazon Redshift Spectrum ✅
Amazon EMR (Apache Spark) ✅
AWS Glue ETL ✅

Enforcement is consistent regardless of service

Whether a user queries via Athena, runs a Glue job, or reads from QuickSight — the same Lake Formation permission rules apply. The access control lives in one place, not embedded in each service.

Key insight: Before Lake Formation, an analyst with Athena access and a data engineer with Glue access might both read columns they shouldn't — because each service enforced only IAM S3 bucket permissions. LF adds a layer that applies uniformly.

Introduction

Feature Areas at a Glance

Lake Formation organizes its capabilities into three pillars, each solving a distinct data governance concern.

Data Ingestion & Management

Bringing data into the data lake, cataloging schemas, labeling resources with tags for access control, and scheduling incremental loads from database sources.

Security Management

Defining and enforcing access controls at database, table, column, row, and cell levels. Includes audit logging, cross-account sharing, ABAC, and hybrid access mode.

Bring Data Into the Catalog (Federation)

Connecting external data sources and Redshift catalogs without migrating metadata. Includes Redshift integration, external metastores, S3 Tables, and AWS Data Exchange.

These pillars overlap in practice — you ingest data into the catalog, then apply security rules, then optionally federate access to it from external systems. All three are centrally managed from the Lake Formation console.

Introduction

Data Ingestion and Management Features

Import data from databases already in AWS

  • Supported sources: MySQL, PostgreSQL, SQL Server, MariaDB, Oracle (in RDS or on EC2)
  • Supports both bulk and incremental data loading
  • Reads data + schema, imports into data lake, records metadata in central catalog

Import from other external sources

  • Connect to on-premises databases via JDBC
  • For unsupported databases, create custom ETL jobs with AWS Glue

Catalog and label data

  • Use AWS Glue Crawlers to read S3 data and extract database/table schema automatically
  • Schema stored in a searchable Data Catalog
  • Manage permissions with Lake Formation tag-based access control (TBAC) on databases, tables, and columns
Crawlers + LF-Tags together are the foundation of a self-healing data lake: new data lands on S3, a crawler discovers its schema, tags are applied via automation, and the security rule written once by security team auto-applies to new assets.

Introduction

Hybrid Access Mode

  • Lets you secure and access cataloged data using both LF permissions AND IAM permissions policies (for S3 and Glue actions)
  • Data administrators can onboard LF permissions selectively and incrementally
  • Focus on one data lake use case at a time rather than a big-bang migration
  • Enables LF permissions for specific users without interrupting existing IAM-only workloads
Migration path: Start with IAM-only. Flip hybrid mode on for new databases. Grant LF permissions to new use cases. Existing workloads see no change. Gradually expand LF coverage.

Grant Models

Named Data Catalog resources vs. Resources matched by LF-Tags — two fundamentally different mental models behind the same Grant button.

Grant Models

Two Ways to Grant in Lake Formation

When you click Grant in the LF console, you must pick a model. Same Grant button — two fundamentally different mental models.

Named Data Catalog Resources

You pick specific things by name. You click on a specific database, table, and optionally specific columns.

  • Static scope — exactly the resources you ticked
  • New tables require a new grant
  • Audit is simple — Data Permissions page shows db → table directly

Resources Matched by LF-Tags

You write a tag expression. LF matches resources at query time — today and in the future.

  • Dynamic scope — defined by predicate, not a name
  • New tables tagged accordingly are automatically covered
  • Audit requires joining tag expression with tag attachments
The conceptual difference: Named grants are like writing a guest list with specific room numbers. LF-Tag grants are like a rule at the door: "anyone with a badge labeled public + finance may enter" — applied to all rooms with those tags, including rooms added after the rule was written.

Grant Models

Named Data Catalog Resources

You pick specific things by name: database = quicklabs_student8_lake, table = curated_oil, optionally specific columns.

Property Behavior
Scope Static — exactly the resources you ticked
New tables created tomorrow Not covered — need a new grant per resource
Audit / listing Easy: Data Permissions page shows db → table directly
Best for Small data lakes, one-off grants, ad-hoc sharing
Named grants are the right default when you have a small number of well-defined resources and don't expect the catalog to grow rapidly. They're easiest to audit and understand.

SQL analogue

GRANT SELECT
  ON quicklabs_student8_lake.curated_oil
  TO student7;

This is exactly the mental model — you're granting permission on a named, specific object. If you create a curated_oil_v2 table tomorrow, student7 cannot read it until you issue a new grant.

When this becomes painful: your data lake has 200 tables and you need to give 15 analysts different column-level access to each table. That's potentially 3,000 grants — and every new table is another set of manual grants.

Grant Models

Resources Matched by LF-Tags

You write a tag expression; LF resolves it at runtime. Example: sensitivity = public AND department = finance — applies to every resource that matches today AND in the future.

Property Behavior
Scope Dynamic — defined by predicate, not a name
New tables tagged accordingly tomorrow Automatically covered — zero new grants required
Audit / listing Harder: must join tag expression with current tag attachments
Best for Large orgs, many tables, consistent classification policy
LF-Tags decouple the access rule from the resource list. Security writes the rule once. Data engineering manages tags on new assets. No ticket to security needed.

Conceptual SQL analogue

GRANT SELECT
  ON ANY TABLE
  WHERE classification = 'public'
  TO student7;

Standard SQL doesn't support this — LF-Tags are the only way to express "grant access to a category of data rather than a named object."

Tag assignment happens separately

A data engineer tags curated_oil with sensitivity=public, department=finance. The grant you wrote on slide 9 now automatically applies — no security intervention.

Grant Models

When the Difference Matters

Scenario Named grants LF-Tag grants
1 table, 1 consumer 1 grant 1 grant + 1 tag attachment
50 tables, 1 consumer, same access 50 grants 1 grant + 50 tag attachments
Add a 51st table next week +1 more grant manually Just tag the new table — grant auto-applies
Audit "what can student7 read?" Look at Data Permissions list — direct Look at tag expression + every table's current tags
Coordination cost Every new table = ticket to security team Data eng tags; security wrote the rule once
The architectural punchline: Named grants couple access control to resource creation. Tag-based grants decouple them. This is the difference between "security is a bottleneck" and "security scales." At 500+ tables, named grants become operationally untenable.

Demo Lessons

The Problem LF Was Built to Solve

Before LF — IAM-only access

  • Every analyst needs S3 read on raw bucket → they see PII columns just by SELECT *
  • Every analyst needs Glue catalog read → they see every table, governed or not
  • Column-level masking? Build it in every query, every view, every dashboard
  • Row-level access? Same — every consumer reimplements the filter
What you're controlling IAM controls LF controls
"Can this user use Athena at all?" ✅ —
"Can this user see column volume?" — ✅
"Can this user see rows where month = 12?" — ✅
"Can this user read S3 at all?" Required, but empty LF issues temp creds at query time
Punchline: IAM stays static per role. LF grants are where access changes. Three users with identical IAM policies can see three different result sets from the same query — and none of them have direct S3 access to the bucket.

How LF Actually Intercepts a Query

LF hands out temporary S3 credentials at query time, scoped exactly to what you're allowed to see.

1
Athena receives SELECT * FROM oil
2
Athena → Glue Catalog: "where is this table's data?"
3
Glue Catalog → Lake Formation: "does this principal have SELECT on this table?"
4
LF evaluates grants — named grants, LF-Tag expressions, data cell filters, all combined
5
LF returns GetDataAccess response — short-lived IAM credential scoped to: allowed columns only, allowed rows only (predicate-pushed), registered S3 prefix only
6
Athena uses those temp creds to read S3 — never the user's own IAM
7
Athena returns rows — already filtered

Key implications

  • User does NOT need direct S3 read on the data lake bucket
  • User CANNOT bypass LF by going to S3 directly — they have no creds
  • The audit trail is single-source: every read is a GetDataAccess event in CloudTrail
Why this matters for compliance: With traditional S3 bucket policies, a determined analyst could copy the IAM credentials from their Athena session and use them to access S3 directly. With LF-vended credentials, those creds are scoped to an exact set of columns and an exact S3 path — useless for anything else.

Same Query, Three Views — The Demo Punchline

SELECT * FROM quicklabs_student8_lake.curated_oil LIMIT 5;
Persona Result
Data Lake Admin All 11 columns, all 6,367 rows
Manager (table SELECT grant) All 11 columns, all 6,367 rows
Analyst (column grant — no volume) 10 columns (volume gone), all 6,367 rows
Analyst with data filter (month = 12) 10 columns (volume gone), ~530 rows
Tagged analyst (sensitivity = public) 10 columns (volume tagged restricted), all rows
Five different grant models, five different result sets — all from the same query string. Governance is declarative (grant a rule) instead of imperative (rewrite the query, build a view, manage S3 prefixes manually). The security boundary lives in LF, not in application code.

Audit: Every Read Produces a GetDataAccess Event

Every time a user's query touches LF-governed data, AWS records a GetDataAccess event in CloudTrail.

Event payload contains

  • The IAM principal (who)
  • The table being read (what)
  • The exact column list LF authorized (after grants + filters + tag rules combined)
  • The Athena query ID (correlate to the SQL itself)
  • Timestamp + source IP

What you can prove

  • "Show me every query analyst X ran on pii_users in March." → CloudTrail filter
  • "Did anyone read column ssn ever?" → filter GetDataAccess where authorized column list contains ssn
  • "Reconstruct exactly what data this analyst could see at any point in time." → cross-reference GetDataAccess with GrantPermissions/RevokePermissions events
vs. traditional S3 access logs: With IAM + S3 logs, you see "Alice read s3://bucket/x.parquet." You don't see which columns were materialized. GetDataAccess captures the exact column list that was authorized — making LF's audit capability qualitatively different from bucket-level logging.
